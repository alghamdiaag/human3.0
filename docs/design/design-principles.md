# Human3.0 Design Principles

## Purpose

This document defines the visual and structural identity of the Human3.0 UI layer.

It exists to ensure that every future screen, component, and landing page remains consistent with the established design language.

This is the design source of truth for the frontend.

---

## Core Identity

Human3.0 is not designed as a generic SaaS dashboard.

It is a focused cognitive environment.

The interface must feel like:
- a structured thinking system
- an editorial decision environment
- a serious, high-attention workspace

It must not feel like:
- a busy analytics dashboard
- a playful consumer app
- a template-based startup UI
- a multi-widget control panel

---

## Design North Star

The design direction is:

**Digital Curator**

Human3.0 behaves like a high-end editorial system for thought.

It filters noise, creates focus, and presents information with deliberate hierarchy.

The UI should communicate:
- clarity
- restraint
- seriousness
- progression
- cognitive depth

---

## Layout Philosophy

### Intentional Asymmetry

The layout should not feel mechanically centered at all times.

Use asymmetry to create:
- tension
- focus
- visual pacing
- editorial rhythm

Main content can sit in a center-right column while labels, metadata, or supporting context live in the left margin.

---

### Visual Pause

Large spacing is not empty waste.
It is part of the reading and thinking experience.

Use generous vertical spacing between major sections so each block feels cognitively distinct.

The interface must breathe.

---

### One Primary Focus Per Screen

Each screen must have one dominant cognitive action.

Examples:
- start
- understand the stage
- answer one question
- review a reflection
- resolve a contradiction
- continue to the next stage
- generate the report
- submit a follow-up reflection

Do not place multiple competing priorities on the same screen.

---

## Surface Philosophy

Human3.0 uses tonal layering instead of heavy structural framing.

### Base Principle
Depth should come from:
- surface shifts
- tonal contrast
- spacing
- restraint

Not from:
- many borders
- card clutter
- heavy separators

### Surface Hierarchy
Use layered dark surfaces to separate levels of importance:
- base background
- recessed areas
- elevated content
- floating controls

---

## The No-Line Rule

Avoid using standard 1px divider lines as the default method of separation.

Prefer separation through:
- spacing
- tonal shifts
- containment
- hierarchy

If a boundary is necessary, it should feel subtle and secondary.

---

## Typography Principles

Typography must feel:
- editorial
- confident
- sharp
- quiet

### Headings
Headings carry authority.
They should be bold, high-contrast, and visually dominant.

### Body Text
Body text should remain highly legible and calm.
It should support deep reading, not fast scanning only.

### Labels
Labels should be small, restrained, and structural.
They provide context, not decoration.

---

## Interaction Philosophy

The interface should feel controlled, not reactive in a chaotic way.

### Primary Actions
Each screen should usually have one primary action only.

### Secondary Actions
Secondary actions are allowed only when necessary:
- go back
- adjust
- skip intro

### Progress
Progress should always be visible when the user is inside the protocol.

### State Changes
Transitions between screens should feel intentional and quiet.
No flashy motion.
No distracting effects.

---

## Component Philosophy

Components must feel like:
- instruments
- structured blocks of thought
- restrained containers

They must not feel like:
- decorative product cards
- consumer-style widgets
- generic component library artifacts

---

## Input Philosophy

Input areas are not simple form fields.
They are thinking zones.

Textareas and response areas should:
- feel focused
- feel quiet
- invite depth
- avoid visual clutter

The user should feel they are writing into a serious cognitive workspace.

---

## Buttons

Buttons should communicate hierarchy clearly.

### Primary Button
Used for:
- continue
- execute
- generate report
- start protocol
- submit reflection

### Secondary Button
Used sparingly for:
- go back
- adjust
- skip intro

Avoid having multiple equally prominent actions.

---

## Navigation

Navigation is structural, not exploratory.

Top navigation and side navigation exist to:
- orient the user
- reinforce stage progression
- preserve system identity

They should not encourage random browsing during the protocol.

---

## Screen Character

Every screen in Human3.0 should preserve these qualities:

- focused
- high-contrast
- spacious
- deliberate
- quiet
- exacting

No screen should feel casual or improvised.

---

## Consistency Rule

Any new page, section, or landing page added later must inherit from these principles.

If a future design idea conflicts with this document, this document wins unless deliberately replaced by a newer system-level decision.

---

## Summary

Human3.0 design is:

- editorial
- restrained
- cognitive
- structured
- asymmetrical
- progression-driven

It is not:

- decorative
- crowded
- dashboard-first
- startup-generic
- visually noisy
